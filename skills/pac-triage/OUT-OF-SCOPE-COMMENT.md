# Writing Out-of-Scope Comments

Out-of-scope decisions are GitHub-native scope-boundary records. They replace Matt's local `.out-of-scope/*.md` knowledge base with issue comments plus the `out of scope` label.

Use this outcome when the request crosses a durable project boundary. This is different from `wontfix`:

- **`out of scope`** — the request does not belong in the project; future similar requests should usually get the same answer unless scope changes.
- **`wontfix`** — the request may be in scope or valid, but this specific issue will not be actioned.

Every out-of-scope comment must start with the machine-readable marker and triage disclaimer:

```markdown
<!-- pac:out-of-scope -->
> *This was generated by AI during triage.*
```

## Principles

- Preserve durable reasoning, not temporary prioritization.
- Explain the project boundary being protected.
- Mention maintenance cost, product philosophy, technical constraints, or strategic decisions when relevant.
- Offer escape hatches or alternatives when they exist.
- Link prior similar requests so the issue tracker becomes the knowledge base.
- Do not create or update `.out-of-scope/` files.

## Template

```markdown
<!-- pac:out-of-scope -->
> *This was generated by AI during triage.*

## Out-of-Scope Decision

**Outcome:** out of scope
**Label:** `out of scope`
**Request:** one-sentence summary of what was requested

**Decision:**
Clear statement of what mypac will not support or pursue.

## Why this is out of scope

Durable reasoning. Explain the project boundary, maintenance cost, product philosophy, technical constraint, or strategic decision. Avoid temporary reasons like "not now" or "too busy".

Useful prompts for this section:

- What project boundary does this request cross?
- What maintenance surface would this add?
- What existing workflow already covers the need?
- What tradeoff would become worse if this were accepted?
- Why would future similar requests get the same answer?

## Alternatives / Escape Hatches

Optional. Explain what the requester can do instead.

- alternative 1
- alternative 2

## Related or Prior Requests

Optional but recommended. Link issues, PRDs, ADRs, or previous out-of-scope decisions.

- #123 — short title

## Reconsideration Criteria

What would need to change for this to become in scope? If the decision is expected to be permanent, write `None expected`.
```

Omit optional sections only when they would add no useful context.
